### Key finding — brochure vs policy_doc are not redundant
policy_doc has 3-4x the section count of brochure (64 vs 17 sections on
New Tech Term) and includes a full Definitions block absent from brochure.

**Merge rule:** policy_doc is authoritative for every field. Brochure is
used only to supply what policy_doc doesn't carry — confirmed so far to be
just the sample illustrative premium table. One JSON record per policy,
track field-level provenance (policy_doc vs brochure) so conflicts, if any
turn up, are traceable. Not yet diffed for actual value conflicts between
the two documents on overlapping fields — do this before building merge
logic.

### Key finding — term plans split into 3 structural variants
- **Standalone** (Saral Jeevan Bima, Yuva Term, Digi Term): fixed sum
  assured, no rider compatibility, some offer death-benefit-in-instalments.
- **Credit life** (Yuva Credit Life, Digi Credit Life): sum assured
  *decreases* over the term, tied to an outstanding loan schedule — "pure
  decreasing Term Assurance plan" per the plan's own brochure language.
- **Rider-enabled** (New Tech Term, New Jeevan Amar): explicitly offer an
  optional Accident Benefit Rider (own UIN). The other 5 plans mention
  "rider" only in generic Section 45 legal boilerplate — not real
  compatibility. A naive text-match extraction rule would misfire on this.

No ROP (Return of Premium) variant exists in this term set — all 6 pay
zero maturity benefit on survival. Verified, not assumed (initial "ROP"
grep hit was a false positive matching "proposal").

### Layer 2 schema — normalized decision layer (applies across all categories)

Derived from Layer 1 at ingestion time (same Gemini call, second reasoning
pass — not a separately extracted field set). Group C is a direct copy of
Layer 1 bounds. Group B restructures Layer 1's payout language into a
controlled vocabulary. Group A is the only genuinely interpretive layer.

    Group A — concern_tags (array):
      "income_replacement", "debt_linked_cover", "child_education_fund",
      "retirement_income", "estate_legacy_planning",
      "forced_savings_discipline", "medical_critical_illness_addon",
      "liquidity_via_policy_loan"

      Deliberately excludes a tax-benefit tag — verified via grep across all
      10 documents checked that tax mentions are identical boilerplate
      ("consult your tax advisor") with zero discriminative power.

      No "other" catch-all in this group — unlike Group B, these tags are
      interpretive judgments, not facts extracted from printed text. An
      "other" bucket here fails silently (a policy just never matches a
      concern-based search) rather than surfacing for review. The correction
      mechanism is manual: read documents, find a real recurring concern the
      tags miss, add it as a named tag (this is how liquidity_via_policy_loan
      was found this session).

    Group B — payout mechanics (arrays — confirmed necessary, real plans
    combine values, e.g. Jeevan Umang pays periodic survival benefit AND
    lump sum at maturity simultaneously):
      payout_on_death: ["lump_sum" | "instalments_available" |
                         "decreasing_schedule" | "other"]
      payout_on_survival: ["none" | "lump_sum_at_maturity" |
                            "periodic_survival_benefit" | "bonus_accrual" |
                            "other"]
      payout_notes: string | null   // required if "other" present above
      is_participating: boolean
      builds_cash_value: boolean
      cash_value_loan_available: boolean
      cover_basis: "fixed" | "decreasing_loan_linked"

    Group C — deterministic pre-filter facts (direct copy from Layer 1):
      min_age, max_age, min_sum_assured, max_sum_assured, min_term, max_term
      compatible_riders: []

### Third layer — chunk-level narrative embeddings
Distinct from both Layer 1 and Layer 2. Structure-aware chunks of the raw
policy_doc text (PART/section headers), embedded via voyage-law-2, tagged
with policy_id/category/section_name. Used only by step 5 (semantic
retrieval), not the deterministic filter. Layer 1 feeds this indirectly
(same source document) but chunking/embedding is a separate ingestion
sub-step from Layer 1/2 extraction.

### Query-time filter logic (resolved this session)
- Step 3 filter: single OR-pass on Group A concern_tags (match-at-least-1),
  not sequential AND-then-relax queries.
- `concern_match_count` (0 to N selected concerns matched) used as PRIMARY
  sort key. Reranking's semantic score used as tiebreaker WITHIN each
  count-block, not blended into one weighted number. Rationale: avoids
  inventing an arbitrary weighting formula, keeps ranking explainable for
  the eventual LLM judge / golden-set evaluation.
- Existing eligibility-fallback rule ("fewer than 3 matches, relax
  constraints") applies to Group C only. Concern-matching does NOT get a
  separate relaxation pass — this was an identified risk (two independent
  relaxation mechanisms firing on the same query) and is resolved by not
  having concern-matching hard-filter at all; it's a sort key, so nothing
  needs relaxing.

### Price/premium (new, not in original architecture doc)
- Problem identified: nothing in the schema penalizes cost, so
  concern_match_count + semantic relevance alone will systematically favor
  feature-rich, expensive policies.
- Resolution: NEW pipeline step, between deterministic filter (step 3) and
  reranking (steps 5-6). For each candidate surviving the eligibility
  filter, compute an estimated premium via LINEAR interpolation from that
  policy's brochure sample-premium table, against the user's actual
  age/sum-assured/term (not a fixed baseline — user-input-based, per
  decision this session).
- Used as a THIRD independent scoring signal, separate from
  concern_match_count and semantic score — same non-blended pattern.
- Edge case: user profile falls outside a policy's sample table range
  (e.g. age above what the table covers) → EXCLUDE that candidate from
  results, log why, continue the query. Do not fail the whole query.
- Explicitly UNVALIDATED assumption, deferred: (1) whether premium curves
  are actually close to linear across sample points — not checked; (2)
  whether Gemini can reliably interpolate at all — assumed "decent" for
  v1, to be validated later against the golden set. Both listed in
  architecture.md's "Verify before building" section as open items, not
  silently assumed.
- Step 7 (narrative generation) must disclose shown premiums are
  reference estimates, not exact quotes.
