# Progress Log — 2026-07-12

## Term-assurance extraction testing (Layer 1)

Ran iterative extraction testing against 2 policies: Saral Jeevan Bima
(859, standalone variant) and Yuva Credit Life (877, credit_life variant),
using Gemini `flash-lite-latest`. Fixed 13 extraction traps in the prompt
over the course of testing:

1-4. Rider compatibility (UIN-only, not generic Section 45 mentions), tax
   boilerplate exclusion, nullable-field handling, unverified-field flags
   (pre-existing, from 2026-07-07 session).
5. Multi-column premium tables (Single/Regular/Limited PPT) → separate
   array rows, not collapsed into one value.
6. Underwriting exception clause on a stated Sum Assured max ("may be
   considered... subject to Reinsurer decision") does not make the max
   null — the general rule's number is still real, exception goes in notes.
7. "No surrender value... however [conditional refund] shall be payable"
   → `surrender_value_applicable: false`, refund goes in
   `policy_cancellation_value_formula`, not conflated.
8. `"%o"` after a decimal in a rebate context is a corrupted per-mille (‰)
   rendering artifact — keep literal, flag via structured note.
9a/9b. Multi-table detection: documents can have two sample-premium tables
   back-to-back under ONE shared heading with no visual break — the only
   signal is a new sentence restating a different Sum Assured/Term.
   Two-pass "inventory then extract" prompting added after finding the
   model would silently drop the second table.
10. Redline/tracked-change artifacts (two adjacent numbers, e.g. "15 30
    days") — flag both via notes, don't silently pick one.
11. Free Look Period vs. a separate refund-processing timeline — don't
    conflate into the same field.
12. `premium_payment_option: "single"` must always pair with
    `payment_mode: "lump_sum"`.
13. A stated Sum Assured MINIMUM can have a lower conditional exception
    gated on age band + purpose (found on Yuva Credit Life: general min
    50L, conditional 20L for ages 21-45 tied to a housing loan) — same
    shape as trap 6 but for the minimum.

### Input method: PDF-native vs. pdftotext text

Same prompt, same model, tested on 859 (which has the back-to-back
shared-heading table pattern from trap 9a/9b):
- **Text input** (`pdftotext -layout` extracted `.txt`, substituted into
  prompt placeholders): inconsistent even after the trap 9 fix — 3 of 6
  runs found both premium tables (44 rows), 3 of 6 missed the second
  table (40 rows only). Roughly a coin flip.
- **PDF input** (native file upload via Gemini Files API, attached
  directly to the prompt): 7 of 7 runs clean — 4/4 on 859 (both tables
  every time), 3/3 on 877 (byte-identical output every run).

Conclusion: PDF-native input is materially more reliable than
`pdftotext`-flattened text for this failure mode, at the same model cost.
Promoted PDF input as the production approach (`docs/prompts/prompt_a_pdf.txt`),
moved the text-mode prompt to `docs/prompts/appendix/prompt_a_text_mode_deprecated.txt`.

### Model choice: Flash-Lite vs. Pro

## Layer 2 derivation testing

### First version: PDF-grounded (reverted)

Built `prompt_b_pdf.txt` + a new script attaching policy_doc + brochure
PDFs to the Layer 2 call, on the assumption Group A's concern-tag
reasoning needed brochure "positioning" language Layer 1's schema
doesn't capture.

Found a real bug: on 877, the model sometimes re-derived
`group_c.min_sum_assured` directly from the attached PDFs instead of
copying Layer 1's already-resolved value (5000000, the correct general
rule per trap 13) — one run picked 2000000 (the conditional exception
value) instead, with `derivation_notes` explicitly showing the model's
reasoning process re-litigating the same general-rule-vs-exception
ambiguity Layer 1 had already fixed. Added an explicit "copy Group C
verbatim, never re-derive from the PDFs" instruction; this improved
consistency (2/3 clean after the fix) but didn't fully eliminate the risk.

### Second version: Layer1-JSON-only (adopted)

User asked directly: does Layer 2 need the raw docs at all, given it's
just tagging + remapping Layer 1's own output? Tested a version of
`prompt_b.txt` that takes ONLY the Layer 1 JSON, no PDFs, no `pdftotext`.
4/4 runs (2 per policy) produced identical `group_a_concern_tags`,
`min_sum_assured`, and `cover_basis` to the PDF-grounded version — no
measurable benefit from source-doc grounding, and zero risk of the
Group C corruption bug, since there's no source text to re-derive from.

Promoted this as the production approach. Moved both prior variants
(PDF-grounded, original text-mode) to `docs/prompts/appendix/` as
deprecated reference, each with a note explaining why superseded.

This also corrected an earlier architecture assumption: the original
plan assumed Layer 2 ran as a second reasoning pass in the same Gemini
call as Layer 1. Testing confirmed it works better as a fully separate
call, given only Layer 1's JSON.

## Pipeline scripts

Iterated through a few shapes before settling:
1. Two scripts (`run_extraction.py` text-substitution, `run_extraction_pdf.py`
   PDF-attachment) — the original shape.
2. Merged into one script with argument-shape dispatch (`key=path` =
   substitute, bare path = attach) — turned out to be solving a
   generality problem that doesn't exist, since Layer 1 only ever needs
   PDF attachment and Layer 2 only ever needs text substitution.
3. Split back into two clearly-named scripts matching each stage:
   `run_layer1_extraction.py` (PDF-only) and `run_layer2_derivation.py`
   (text-substitution-only). Settled here.
4. Added `run_pipeline.sh` to orchestrate both stages with output
   validation (file exists, non-empty, valid JSON) between steps.
5. Added explanatory comments to all three scripts — the code encodes
   non-obvious decisions (PDF vs text, why Layer 2 gets no source docs,
   `set -a`/`set +a` semantics) that wouldn't be visible from the code
   alone on a later read.
